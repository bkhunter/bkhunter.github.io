Blogface

January - April 2016

Have you ever wanted to connect with someone who uses Google+ even though you use Facebook? With Blogface this problem is solved: you can blog, view posts, share, and connect even if everyone you know is using another service. Created in a team of five for a Web Applications and Architecture course at the University of Alberta.

Overview

Blogface was a group project with the goal of making a fairly complex website using freely available tools. There were two sides to this website, the first being a fairly straightforward implementation of a blogging/social network style application. This included things like blogging, making friends, adding users, viewing and sharing posts etc. The second side was that we had to interact with other group’s applications and enable users on our site to connect with users on their site, display posts made on their site on our site, and all the necessary details in between. This made is necessary for groups to create a RESTful api to give teams a consistent and systematic way of connecting with each other. Blogface was a great learning experience since failure to work and plan with multiple teams, would spell disaster for the project.

To develop Blogface we used the Django web framework to handle the backend (including the Django Rest Framework for the api - pictured here), and made the frontend using HTML, CSS, Node.js, and AngularJS. The website was initially going be hosted with Heroku, a free web hosting service, but due to better free account features we switched to OpenShift. My contribution was mostly on the back end, which included creation and testing of the API, as well as working with Django, PostgreSQL, and our web hosting services. My group took a different approach to development in that we ‘ate our own dog food’, meaning that our back end and front end were developed and hosted distinctly. This forced us to make calls to our own API as opposed to directly querying our database, and ensured the API functioned as required when other group needed to use it. Agile development was suggested, but other than requiring unit testing, following user stories, and developing in sprints, it wasn’t a large factor in this project.
